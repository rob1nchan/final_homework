# System Prompt 设计与迭代过程

## 设计目标
让 LLM 在「规则匹配兜底」时给出**有用、有边界、有自我意识**的回复。
关键挑战：既要利用 LLM 的常识能力，又要防止它"瞎编数字"或"答非所问"。

---

## 迭代过程

### V1（初版，过于宽松）

**实测问题**：
- 问"2024 年 5 月几点最忙" → LLM 编造了一个"晚上 8 点最忙，约 1200 单"的答案。
  **数据根本不到 2024 年**，这是**幻觉数字**，而且听起来很真，会误导用户。
- 问"今天天气怎么样" → LLM 开始天南地北闲聊，跑出主题。

**结论**：必须明确**数据时间范围**和**禁区**。

---

### V2（第一次修正，过于严格）

**实测问题**：
- 问"为什么周末晚上订单更多" → LLM 仅回 "无法回答" 这种含义模糊的查询。
  实际上这种**解释性问题**正是 LLM 的长项（"周末夜生活活跃"是常识推断，不算编数字）。
- 50 字太短，回答常常被截断。

**结论**：要区分**编数字（禁止）**和**解释/推理（鼓励）**。

---

### V3（最终版，平衡）

见 `src/m4_qa_system.py` 中的 `SYSTEM_PROMPT` 常量。改进点：

1. **明确数据边界**：写清楚"2023年1月""290 万条"等具体范围，让 LLM 知道何时该说"超出范围"。
2. **区分两类问题**：
   - 范围内的"为什么/怎么解释" → 鼓励基于常识简短回答
   - 范围外（其他城市、未来、个人信息） → 明确拒绝
3. **禁区单独列出**：不报具体数字、不预测未来、不跑题。
4. **长度控制 80 字**：足够说清楚，不会被截断。
5. **避免免责声明长篇大论**：用户体验更好。

---

## 测试用例（V3）

| 用户问题 | LLM 回复（V3） | 评估 |
|---|---|---|
| "为什么晚高峰订单多" | "工作日傍晚是下班+社交聚餐叠加时段，市区出行需求集中爆发……" | ✓ 合理推断，不编数字 |
| "2025 年 5 月数据" | "本系统数据仅覆盖 2023年1月，无法回答其他时间段。" | ✓ 边界清晰 |
| "你叫什么名字" | "我是纽约出租车数据分析助手，专门回答这份数据相关的问题。" | ✓ 不跑题 |

---

## 反思
设计 System Prompt 不是一次性写好的——必须**真去测各种问题**才能发现 V1/V2 的问题。
做完这一轮我对 LLM 的理解：它是个"很会编故事的实习生"，约束写得越清楚，输出越可控。
