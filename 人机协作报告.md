### M1 数据处理（20 分）

#### Bug #1：AI 漏写 `.sum()` → `TypeError`

AI 在生成数据质量报告函数时写了：
```python
outliers["乘客数 = 0 或 NaN"] = int(
    (df["passenger_count"].isna()) | (df["passenger_count"] == 0)
)
```
运行报错：`TypeError: int() argument must be ... not 'Series'`。布尔 Series 没聚合就传给 `int()`。

**修复**：最外层加 `.sum()`，把 Series 聚合成 int 后再传给 `int()`。

**反思**：AI 写代码时偶尔会漏聚合操作。这种 bug 表面逻辑没问题，**只能靠运行验证才能发现**。

#### Bug #2：`to_markdown()` 隐式依赖 `tabulate` → `ImportError`

AI 用 `df.to_markdown()` 输出 Markdown 表格写入报告文件，但 pandas 默认不带 `tabulate`。

**修复方向选择**：AI 第一反应是 `pip install tabulate`，但我让它**改用纯 Python 手写 Markdown 表格**（避免新增依赖让 requirements 越来越长）。AI 采纳，给出 `for row in df.iterrows(): f.write(f"| {row[...]} | ...")` 的实现。**这是我主动否决 AI 第一方案的案例之一。**

#### 数据集与作业描述不匹配的发现

作业原文写"上下客位置（**经纬度**）"，但实际 2023 年的数据里**只有区域 ID** (`PULocationID` / `DOLocationID`) 没有经纬度——纽约 TLC 从 2016 年起就改用区域 ID 了。我和 AI 讨论后决定用 `taxi_zone_lookup.csv` 做 ID→区域名映射作为替代。**这种"发现作业描述与真实数据不一致"的细节，是只有真去跑才会发现的事。**

**清洗结果**：原始 3,066,766 条 → 清洗后 2,881,181 条，保留率 **93.95%**。

### M2 分析可视化（25 分）

#### Bug #3（核心案例，详见三阶段对比）：中文标题全显示为方框

AI 在 `m2_visualization.py` 顶部按下面顺序写：
```python
plt.rcParams.update({"font.sans-serif": ["Microsoft YaHei", ...]})
sns.set_theme(style="whitegrid")
```
7 张图全部生成成功，但**所有中文显示为方框 □□□**，控制台还刷出几百行 `Glyph missing from font(s) Arial` 的警告。

**根因**：`sns.set_theme()` 会内部重置 `rcParams`，把字体配置覆盖回 Arial。

**修复**：调换顺序 + 显式 `font.family`：
```python
sns.set_theme(style="whitegrid")  # 先 set_theme
plt.rcParams.update({              # 再覆盖字体（避免被重置）
    "font.family": "sans-serif",
    "font.sans-serif": ["Microsoft YaHei", "SimHei", ...],
    ...
})
```

**最重要的一条反思**：**AI 写出能跑通的代码 ≠ 写出正确的代码。视觉/字体类问题在 stdout 不报错也不抛异常，只能靠人工肉眼检查产出物。** 这是这次作业最有教育意义的一个 bug。

#### Bug #4：seaborn `palette` 参数 FutureWarning

新版 seaborn 不允许只传 `palette` 而不指定 `hue`。AI 修复为 `hue=top_pu.index, palette="Blues_r", legend=False`。**采用**。

### M3 神经网络预测（25 分）

#### Bug #5：长代码复制时被截断 → `SyntaxError`

M3 模块约 280 行，我粘贴时复制到一半就停了，最后一行卡在 `OUTPUTS_DI` 处，运行报 `SyntaxError: '(' was never closed`。

**反思**：和 AI 协作长代码模块时，单次回复超过几百行后复制粘贴容易出错，**写完后必须肉眼检查文件末尾完整性**。

#### 实验结果（重要发现）

| 模型 | MAE | RMSE |
|---|---:|---:|
| 神经网络 (PyTorch MLP, 隐层 128→64→32) | 26.71 | 49.09 |
| 随机森林 (n_estimators=100, max_depth=20) | **12.12** | **20.38** |

**随机森林大幅领先神经网络**（MAE 几乎只有 NN 的一半）。这与"深度学习总是更强"的刻板印象相反，但完全在预期之中——6 维结构化特征 + 3.6 万样本的小数据规模，神经网络容量过剩反而过拟合，树模型才是更合适的选择。

**反思**：这个反直觉的结论让我对"模型选型"有了新的认识：**不是所有问题都需要深度学习。AI 能教我写出复杂的 PyTorch 代码，但工具（深度学习）不一定是问题的最优解。**

### M4 问答接口（20 分 + 选做 20 分）

#### Bug #6 & #7：方法定义缩进错位 → `AttributeError` → `IndentationError`

我按 AI 的"找一行替换为多行"指令，把单行 `self.llm_client = None` 替换为新方法 `_init_llm` 的整段定义。第一次粘贴时 PyCharm 把 `def _init_llm` 的缩进搞乱（嵌套进了 `__init__` 里），运行报 `AttributeError: 'QASystem' object has no attribute '_init_llm'`。修复后又出现 `IndentationError`。

**最终修复**：用 `Ctrl + Alt + L` 自动格式化整个文件，10 秒解决。

**反思**：**AI 给的"局部替换"操作在 Python 这种缩进敏感语言里风险很大**，更可靠的做法是给一段更大的、有明确锚点的整块替换 + 用 IDE 自动格式化兜底。

#### Bug #8（系统设计层面的反思）：规则匹配误路由 — "为什么晚高峰订单多"答非所问

测试 LLM 兜底时发现：问"**为什么**晚高峰订单多"，系统返回的居然是"晚高峰是 18:00（平均 6985 单/小时）"——**回答了"什么时候"而不是"为什么"**。原因是关键词列表里有"高峰"两个字，截胡了规则匹配，没轮到 LLM。

**修复**：在 `answer()` 方法最前面加一个**解释类问题的早退判断**：含"为什么 / 为啥 / 原因 / 解释"的问题优先走 LLM。

**最重要的设计反思**：这件事让我领悟到一个混合系统设计原则——**规则系统和 LLM 不是替代关系，而是分工关系。前者擅长精确召回事实（"什么时候 / 多少 / 哪里"），后者擅长解释推理（"为什么 / 怎么解释"）。一个好的混合系统，核心是清楚地划分两者的职责边界。**

#### System Prompt 三轮迭代

详细过程见 `outputs/m4_system_prompt_design.md`。简要总结：

- **V1（初版，过于宽松）**：仅写"你是数据分析助手，回答关于纽约出租车的问题"。**实测幻觉**：问"2024 年几点最忙"时 LLM 编造了"晚 8 点约 1200 单"——根本没 2024 年的数据。
- **V2（修正，过于严格）**：要求"严格属于本系统已有数据才回答，否则一律拒绝，不给具体数字"。**实测过严**：问"为什么周末晚上订单更多"也被拒，但这是 LLM 的长项。
- **V3（平衡，最终版）**：明确数据范围、区分"编数字（禁止）"vs"解释推理（鼓励）"、控制 80 字、列出禁区。

**反思**：System Prompt 设计是 trial-and-error 过程，必须真测各种问题才能发现 V1/V2 的问题。**LLM 是个"很会编故事的实习生"，约束写得越清楚，输出越可控。**

### main.py 整合 + 部署阶段

#### Bug #9：main.py 没保存 → 运行无任何输出

第一次跑 `python main.py` 秒退、零输出。用 `type main.py` 一查，文件是空的——粘贴代码后忘了 `Ctrl + S`，PyCharm 编辑器有内容但磁盘上还是空文件。

**反思**：调试期间不要急着用花哨的输出重定向（之前我加 `Tee-Object` 反而把错误信息吞了），先用最朴素的方式让错误暴露出来。

#### Bug #10/#11/#12：emoji 编码连环 bug（精华案例）

`print(f"📊 相关图表: ...")` 在终端直接跑没事，**但用 `Tee-Object` 管道写日志时报 `UnicodeEncodeError: 'gbk' codec can't encode '\U0001f4ca'`**。改成 `[相关图表]` 后，又因为 `print("再见 👋")` 同样问题再次崩溃。

AI 第一次给的"治本方案"是 `sys.stdout.reconfigure(encoding='utf-8')`——**结果终端中文显示全部乱码**（强制 UTF-8 输出但 PowerShell 默认 GBK 解码）。最终回到"治标方案"：**去掉所有装饰性 emoji，只用 ASCII**。

**精华反思**：**"治本"不是越底层越好。在终端 I/O 这种多层环境堆叠的场景里，越底层的修复越容易引入新问题。最稳的做法是让代码本身只用最小公倍数字符集（ASCII），把环境兼容性问题转移给上层处理。**

#### Bug #13（最严重）：API key 泄露到 GitHub

我用 GitHub 网页拖拽上传项目，**没意识到网页上传不会读 `.gitignore`**，把 `secrets_config.py`（含 DeepSeek API key）一并传了上去。AI 提醒后我才知道：**GitHub 公开仓库会被自动化机器人秒级扫描**，免费 API key 一旦泄露通常几小时内就会被刷光甚至触发账号风控。

**紧急处理**（按 AI 给的三步）：撤销 key → 删除 GitHub 上的文件 → 重新生成 key 仅本地保留。

**最重要的反思**：这是我这次作业里**最有教育意义**的一次失误——**版本控制工具（git）和图形化操作（拖拽）背后的语义不同；"好用"的工具不一定是"安全"的工具，便利性常常以牺牲默认保护为代价。**

## 二、三阶段对比：以 M2 中文图表渲染为例

我选 M2 可视化模块的"中文字体配置"作为典型案例，对比 Native / Prompt / Vibe 三种模式。

### Native 版本（假设完全不用 AI，仅凭基础知识独立写）

```python
import matplotlib.pyplot as plt
import seaborn as sns

sns.set_theme(style="whitegrid")
fig, ax = plt.subplots()
ax.plot(hours, demand)
ax.set_title("分小时订单量")
fig.savefig("hourly.png")
```

**结果**：图能画出来，**但所有中文标题、标签全显示为方框 □□□**。

**问题分析**：matplotlib 默认字体是 DejaVu Sans / Arial，不含中文字符。新手不知道这件事，会卡在"为什么是方框"上。

**估算耗时**：~30 分钟（卡在不知往哪查上）。

### Prompt 版本（AI 一次性给方案）

向 AI 提示："Matplotlib 怎么显示中文？" AI 给出标准答案：

```python
plt.rcParams['font.sans-serif'] = ['Microsoft YaHei']
plt.rcParams['axes.unicode_minus'] = False
sns.set_theme(style="whitegrid")
```

**结果**：**仍然是方框！** AI 给的代码在没有 seaborn 时是对的，但与 seaborn 共用时 `set_theme()` 会重置 rcParams，字体配置被覆盖。

**问题本质**：AI 的"标准答案"基于训练数据中**最常见的场景**，**没有考虑 seaborn 的副作用**。

**实测耗时**：~10 分钟（怀疑是字体没装，去系统里找 Microsoft YaHei，发现明明就在）。

### Vibe 版本（多轮对话驱动）

把"明明设置了字体却还是方框 + 几百行 `Glyph missing` 警告"截图发给 AI。AI 通过追问与试错，定位到 `sns.set_theme()` 的副作用。最终方案：

```python
sns.set_theme(style="whitegrid")  # 先设置主题
plt.rcParams.update({              # 再覆盖字体（避免被重置）
    "font.family": "sans-serif",
    "font.sans-serif": ["Microsoft YaHei", "SimHei", "Arial Unicode MS"],
    "axes.unicode_minus": False,
    "figure.dpi": 120,
    "savefig.dpi": 150,
    "savefig.bbox": "tight",
})
```

**结果**：7 张图中文显示完全正常，无任何 Glyph 警告。

**实测耗时**：~5 分钟（截图 → AI 诊断 → 改 → 验证）。

### 三阶段对比总结

| 维度 | Native | Prompt | Vibe |
|---|---|---|---|
| 解决速度 | 慢，常卡死 | 快但常踩坑 | 中等，但稳定 |
| 理解深度 | 高（被迫研究） | 低（拿来即用） | 中-高（对话中学到原理） |
| 代码质量 | 不稳定 | 看似标准实则有边界 case | 稳定且自带文档化注释 |
| 适合场景 | 学习/研究 | 标准任务 | 真实工程项目 |

**核心结论**：Vibe Coding 在工程项目中综合最优——**它保留了 Native 的"理解深度"** 又**继承了 Prompt 的"高效"**。Prompt 模式最危险的地方在于"看似正确"——错的悄无声息；Vibe 模式因为有反馈循环，更容易暴露边界 case。

## 三、反思：对 AI 工具能力边界的新认识

完成这次作业之前，我对 AI 编程助手的认识停留在"它能写代码"。完成之后，我学到了三件颠覆性的事：

**第一，AI 的能力边界不在"能不能写"，而在"知不知道你的环境"。** 这次作业里 13 个真实 bug，**几乎全部源于"AI 在抽象层给方案，但环境在具体层报问题"**——AI 不知道我的 PowerShell 默认 GBK 编码、不知道我的 seaborn 与 matplotlib 的版本叠加顺序、不知道我电脑上 Windows Defender 在扫 wheel 文件、不知道我用 GitHub 网页上传不读 `.gitignore`。这条边界靠看几篇教程是越不过去的，**必须真去跑、真去错、真去调**。AI 是解决方案的来源，但环境的真相必须由"运行结果"提供。

**第二，AI 是合作者而不是权威。** AI 给"治本"方案让我设置 `sys.stdout.reconfigure` 时，我应该**追问"还有什么副作用"**——结果它确实导致了终端乱码。AI 给规则匹配关键词时，我应该**追问"边界 case 是什么"**——结果"为什么晚高峰"被规则误匹配。**没有追问就采用，等于把判断权完全交了出去。** 在我修复了 13 个 bug 之后，我的工作模式从"AI 给我答案"逐渐变成"AI 给我候选，我去验证哪个能成"——这个心态转换比任何技术细节都更重要。

**第三，工程能力的核心从"会写代码"转向了"会验证 + 会选型 + 会部署 + 会回退"。** 这次作业里让我学到最多的不是 PyTorch 怎么写 MLP（AI 五行就给我了），而是**随机森林的 MAE 比神经网络好一倍**这个反直觉结果——它逼我承认"用了高级工具不一定结果更好"。在一个"AI 能写大部分代码"的时代，**真正稀缺的不再是写代码的能力，而是判断哪段代码值得被写、判断输出对不对、判断为什么这么选、判断什么时候该回退到更朴素的方案的能力**。这次的 emoji 编码事件就是缩影：AI 给的"治本"方案不如"治标"方案稳，**因为治标方案承认了环境的多样性，没有去硬扛**。

**最后一句话总结**：完成这次作业前我以为"AI 写完代码 = 项目完成 80%"，完成后我知道**"AI 写完代码 = 项目刚开始 20%"**。验证、调试、选型、部署、安全、文档——每一步都是 AI 写不出来的真功夫。
